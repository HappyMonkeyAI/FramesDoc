## Why it fits

A system that detects key frames, reads OCR-visible content, combines that with transcript context, and turns the result into structured documentation with timestamps is a clear **Work & Productivity** or possibly **Developer Tools** submission. ## Product shape

A good version of this project would have a pipeline like this: ingest meeting video, generate transcript, extract candidate keyframes, run vision/OCR on frames, ask GPT-5.6 to decide which moments are documentation-worthy, and output a structured page with sections like “Setup,” “Commands shown,” “Warnings,” “How-to steps,” and “Watch this clip.” The timestamp-linking idea is especially good because it turns the result from a passive summary into a navigable knowledge artifact. Framed well, this is not “meeting summarization”; it is “documentation generation from ephemeral team knowledge,” which is sharper and more memorable. [openai.devpost](https://openai.devpost.com/)

## Implementation view

The `video-keyframe-detector` repo is useful as a lightweight starting point because it extracts representative frames from video using frame-difference peak detection rather than trying to solve the whole problem for you. `memvid` looks more like a portable memory/retrieval layer for AI agents, so it could help if you want searchable long-term storage of extracted meeting artifacts, but it is not the core extraction engine by itself. For a hackathon, the leanest build is probably: keyframe extraction, transcript segmentation, OCR plus vision on selected frames, GPT-5.6 ranking/classification of useful moments, then export to Confluence-flavored markdown or HTML. [github](https://github.com/joelibaceta/video-keyframe-detector)

## MVP plan

Build the smallest believable loop: upload one meeting video, generate transcript, extract candidate keyframes, run OCR or vision on those frames, have GPT-5.6 classify which moments are documentation-worthy, then output a clean page with screenshots, extracted text, summaries, and timestamp links back into the video. That gives you a concrete “before and after” demo judges can understand in under a minute, which matters because the required demo video must be under 3 minutes and clearly explain what you built plus how Codex and GPT-5.6 were used. [openai](https://openai.com/index/gpt-5-6/)

A realistic 5-part MVP:
- **Input:** One uploaded MP4 and one-click processing.
- **Extraction:** Use a keyframe detector to pull representative visual moments from the recording. [github](https://github.com/joelibaceta/video-keyframe-detector)
- **Understanding:** Combine transcript chunks with OCR or frame understanding, then use GPT-5.6 to label moments like “terminal command,” “setup step,” “important decision,” or “demo workflow.” [openai](https://openai.com/index/gpt-5-6/)
- **Output:** Generate markdown/HTML or a Confluence-ready page with section headings, screenshots, key commands, and timestamps.
- **Retrieval:** Add simple search across extracted artifacts so “wasn’t that in the video?” becomes a fast lookup instead of scrubbing footage. [github](https://github.com/memvid/memvid)

For the demo, use one especially visual recording — for example a meeting where someone shares a terminal, explains a deployment step, or walks through a new tool — because your value is highest when the important knowledge is both spoken and shown on screen. A weak demo would be a talking-head meeting with little on-screen content, because then it looks like ordinary meeting summarization rather than the richer documentation workflow you described.
